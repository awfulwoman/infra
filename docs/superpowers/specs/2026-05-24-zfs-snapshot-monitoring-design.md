# ZFS Snapshot & Backup Monitoring

**Date:** 2026-05-24
**Status:** Approved

## Problem

ZFS snapshots on source hosts stopped being created for child datasets (e.g. `fastpool/compositions/homeassistant`) on 2026-01-06 when Sanoid was replaced by the custom `system-zfs-policy` role. The failure went undetected for ~4.5 months. There is also no alerting if belinda's pull cron fails silently.

## Goals

1. Alert if a snapshot timer fails to run on any source host (per interval type).
2. Alert if belinda fails to pull from any client.
3. Alert if snapshots are stale on source hosts (job ran but produced nothing usable).
4. Alert if pulled datasets on belinda are stale.

## Architecture Overview

Two complementary mechanisms:

- **Healthchecks.io dead-man's switches** â€” detect when scheduled jobs stop running entirely.
- **MQTT staleness alerts** â€” detect when jobs run but snapshots are old (e.g. child discovery misconfiguration).

Both are opt-in via feature flags, disabled by default.

## Part 1: Healthchecks.io Dead-Man's Switches

### 1A â€” Source host snapshot timers

**Role:** `system-zfs-policy`

**New variables:**

| Variable | Default | Description |
|---|---|---|
| `system_zfs_policy_healthchecksio_enabled` | `false` | Enable healthchecks.io integration |
| `system_zfs_policy_healthchecksio_api_key` | `{{ vault_healthchecks_rw_apikey }}` | API key (reuses existing vault var) |

**Behaviour when enabled:**

During the Ansible run, the role creates 4 healthchecks.io checks per host using `community.healthchecksio.checks`:

| Check name | Schedule | Grace period |
|---|---|---|
| `{{ inventory_hostname }} - ZFS hourly` | `@hourly` | 90 minutes |
| `{{ inventory_hostname }} - ZFS daily` | `@daily` | 25 hours |
| `{{ inventory_hostname }} - ZFS monthly` | `@monthly` | 32 days |
| `{{ inventory_hostname }} - ZFS yearly` | `@yearly` | 13 months |

The UUID returned by each check creation is registered as an Ansible fact and injected into the corresponding systemd service template via `ExecStartPost`:

```ini
ExecStartPost=curl -fsS --retry 3 --max-time 10 https://hc-ping.com/{{ uuid }}
```

Because the snapshot services use `Type=oneshot`, systemd only runs `ExecStartPost` if `ExecStart` exits 0. A failed snapshot run will not ping â€” healthchecks.io will alert after the grace period expires.

Tags applied to all checks: `autogenerated`, `{{ host_type }}`, `{{ inventory_hostname }}`, `zfs-snapshots`.

### 1B â€” Belinda pull cron

**Role:** `backups-zfs-server`

**New variables:**

| Variable | Default | Description |
|---|---|---|
| `backups_zfs_server_healthchecksio_enabled` | `false` | Enable healthchecks.io integration |
| `backups_zfs_server_healthchecksio_api_key` | `{{ vault_healthchecks_rw_apikey }}` | API key |

**Behaviour when enabled:**

One check is created per client host in `backups_zfs_server_pull_group`:

| Check name | Schedule | Grace period |
|---|---|---|
| `{{ inventory_hostname }} - ZFS pull {{ client }}` | matches `backups_zfs_server_cron_hour/minute` | 6 hours |

The UUID is injected into each `scripts/pull-{{ client }}.sh` as a `curl` call appended after the pull script exits successfully:

```bash
curl -fsS --retry 3 --max-time 10 https://hc-ping.com/$HC_UUID
```

Tags: `autogenerated`, `zfs-backups`, `{{ inventory_hostname }}`, `{{ client }}`.

## Part 2: MQTT Staleness Alerting

### Staleness definition

For each dataset, identify its most granular interval with retention > 0. A dataset is **stale** if its newest snapshot of that type is older than `threshold_multiplier Ă— interval`:

| Interval | Default stale threshold (multiplier=2) |
|---|---|
| hourly | > 2 hours |
| daily | > 48 hours |
| monthly | > 60 days |
| yearly | > 18 months |

Threshold multiplier is configurable via `system_zfs_policy_stale_threshold_multiplier` (default `2`).

### MQTT topic structure

```
homeinfra/monitoring/zfs/<host_name>/snapshots   # source host snapshot status
homeinfra/monitoring/zfs/<host_name>/backups     # belinda pull status
```

Topic prefix is configurable via `system_zfs_policy_mqtt_topic_prefix` (default `homeinfra/monitoring/zfs`).

`mosquitto_pub` is called without authentication credentials, consistent with the existing `system-mqtt2cmd` role which also connects unauthenticated. If the broker requires auth in future, both roles will need updating.

### Payload format

Both topics publish a JSON payload via `mosquitto_pub` (already installed as a dep of `backups-zfs-server`; added as dep to `system-zfs-policy` when MQTT enabled):

**Snapshots topic:**
```json
{
  "ok": false,
  "host": "minipc-8gb-homebrain",
  "checked_at": "2026-05-24T17:00:00",
  "stale": ["fastpool/compositions/homeassistant"],
  "healthy": ["fastpool/compositions", "fastpool/misc/syncthing"]
}
```

**Backups topic (published by belinda after each pull):**
```json
{
  "ok": true,
  "host": "minipc-8gb-homebrain",
  "pulled_at": "2026-05-24T20:00:00",
  "stale_datasets": [],
  "healthy_datasets": ["fastpool/compositions", "fastpool/misc/syncthing"]
}
```

### 2A â€” Source hosts

**Role:** `system-zfs-policy`

**New variables:**

| Variable | Default | Description |
|---|---|---|
| `system_zfs_policy_mqtt_enabled` | `false` | Enable MQTT staleness publishing |
| `system_zfs_policy_mqtt_host` | `mqtt.{{ domainname_infra }}` | Broker hostname |
| `system_zfs_policy_mqtt_topic_prefix` | `homeinfra/monitoring/zfs` | Topic prefix |
| `system_zfs_policy_stale_threshold_multiplier` | `2` | Multiplier for stale threshold |

**Implementation:**

`zfs-snapshot-report.py.j2` gains three new CLI args: `--mqtt-publish`, `--mqtt-host`, `--mqtt-topic`. When called with these, after building the report the script evaluates staleness for each managed dataset (using snapshot timestamps from the existing `get_all_snapshots_grouped()` call) and calls `mosquitto_pub` via `subprocess`.

`zfs-snapshot-report-cache.service.j2` is updated to pass the MQTT args to the report script when `system_zfs_policy_mqtt_enabled` is true. The cache update and MQTT publish happen in a single service invocation â€” no additional timer needed.

`mosquitto-clients` is added to the apt packages installed by the role when MQTT is enabled.

### 2B â€” Belinda post-pull check

**Role:** `backups-zfs-server`

**New variables:**

| Variable | Default | Description |
|---|---|---|
| `backups_zfs_server_mqtt_enabled` | `false` | Enable MQTT staleness publishing |
| `backups_zfs_server_mqtt_host` | `mqtt.{{ domainname_infra }}` | Broker hostname |
| `backups_zfs_server_mqtt_topic_prefix` | `homeinfra/monitoring/zfs` | Topic prefix |
| `backups_zfs_server_stale_threshold_multiplier` | `2` | Multiplier for stale threshold |

**Implementation:**

`zfs-pull-backups.py` gains a `publish_mqtt_status()` function called at the end of each per-host pull. It checks the most recent snapshot timestamps for the just-pulled datasets and calls `mosquitto_pub`. `mosquitto-clients` is already installed on belinda.

## Out of Scope

- Home Assistant sensor configuration (user sets up template sensors from MQTT topics)
- Healthchecks.io notification channel configuration (user configures in HC UI)
- Monitoring the yearly timer (low priority; grace period of 13 months makes this less urgent)
